同步： 在某段代码执行的时候，在没有得到返回结果前，其他代码是无法执行的
异步： 在某一段代码异步调用发出后，这个代码不会立刻得到返回结果，而是在异步调用发出后通过某些回调函数的处理拿到结果，异步函数不会阻塞后面代码的执行。

js 是一门单线程语言，如果代码都是同步执行就有可能造成阻塞，如果使用异步则不会阻塞主线程。

事件(冒泡、捕获、委托)

DOM 标准事件流触发的先后顺序为：

  1. 先捕获再冒泡。即当触发dom事件时，会先进行事件捕获，
  2. 捕获到事件源之后通过实践传播进行事件冒泡

addEventListener的第三个参数

在我们平常用的addEventListener方法中，一般只会用到两个参数，一个是需要绑定的事件，另一个是触发事件后要执行的函数，然而addEventListener还可以传入第三个参数：

element.addEventListener(event, function, useCapture);

在前端AI 对话应用中，我们需要实现大模型的流失输出渲染，请问你会选择SSE 还是WebSocket 来实现这个功能？请说明两者的核心区别、适用场景，以及你的选型依据？

重点

  • 前端实时通信
  • AI 流式输出场景技术选型
  • 方案的边界问题

核心区别

  • 通信
    • 协议：SSE（Server-Sent Events）是单向通信协议，服务器可以向客户端发送数据，但客户端不能向服务器发送数据；WebSocket是双向通信协议，允许服务器和客户端之间进行全双工通信。
    • WS：适用于需要频繁双向通信的场景，如在线聊天、实时游戏等；SSE适用于服务器需要定期向客户端推送数据的场景，如新闻更新、股票价格等。
  • 底层协议 SSE 基于标准的HTTP HTTPS 协议无需升级，ws 走ws wss 协议 ，需要协议升级握手
  • 数据类型支持 WebSocket支持多种数据类型（文本、二进制等），而SSE仅支持文本数据。能够适配模型的文本输出
  • 重连机制，SSE内置自动重连机制（EventSource），WebSocket需要开发者自行实现重连逻辑 心跳机制 断开重连。
  • 使用成本，SSE实现简单，适合单向数据流；WebSocket实现复杂，但适合双向通信需求。 SSE适合AI流式输出场景，WebSocket适合需要双向交互的场景。

  $bp: map-get($breakPoints,$breakName);
  @if(type-of($bp) == 'list') {
    $min :nth($bp, 1);
    $max :nth($bp, 1);
    @media(min-width: $min) and (max-width: $max) {
      @content
    }
  } @else {
    @media (min-width: $bp) {
      @content
    }
  }
}
